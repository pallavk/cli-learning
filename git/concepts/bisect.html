Bisect: binary-search for a bug

Somewhere in the last 400 commits a test started failing. git bisect finds the exact commit that broke it in about log₂(N) steps — nine tests, not four hundred.

You know one good commit (an old release where the test passed) and one bad commit (now). The culprit is somewhere between. Bisect checks out the midpoint, you test it, and your verdict halves the search range. Repeat until one commit remains — the first bad one. It's binary search, with git driving the checkouts.

Why it's so fast. Each test you run doubles the commits you eliminate. 8 commits → 3 tests. 1,000 commits → 10 tests. A million → 20. That logarithmic payoff is the whole point: you never read code, you just answer "good or bad?" a handful of times and git pinpoints the breaking commit.

The session, as commands

# start, and tell git the two ends of the range git bisect start git bisect bad # current commit (C8) is broken git bisect good v1.0 # C1 (a tag/hash) was fine # git checks out the midpoint for you — test it, then verdict: git bisect good # this checkout passed git bisect bad # this checkout failed # …repeat until git prints “<hash> is the first bad commit” git bisect reset # return HEAD to where you started (C8)
It's a real checkout — mind your tree. Each step moves HEAD to a different commit and checks those files out, so commit or stash your work first (bisect refuses if the tree is dirty). You're also in a detached HEAD the whole time — that's expected; git bisect reset reattaches you at the end. If a particular checkout can't be tested (won't build, unrelated breakage), use git bisect skip and git picks a nearby commit instead.

Automate it: git bisect run

The real superpower. Hand bisect a script (or test command) that exits 0 for good and non-zero for bad, and it runs the entire search unattended:

# a script, a test, or a one-liner — exit 0 = good, 1–127 = bad git bisect start HEAD v1.0 # bad good, in one line git bisect run ./test.sh # or directly: git bisect run pytest tests/test_login.py # git checks out, runs, reads the exit code, narrows — fully automatic # special: exit 125 means “can’t test this one” → same as skip

Combined with a reliable test, git bisect run turns "which of these 600 commits broke login?" into a coffee break. It's one of git's most underused tools.

The mental model. Bisect assumes the property is monotonic — good for a while, then bad forever after the breaking commit. That fits most regressions (a bug introduced once and still present). If a bug flickers on and off, bisect can land on a misleading commit; tighten your test so "bad" is unambiguous. And "bad" needn't mean a bug — bisect equally finds the commit that fixed something or changed any testable behaviour (see git bisect terms).
