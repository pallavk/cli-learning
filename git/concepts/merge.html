How git merge really works

Merge does one thing rebase never does: it joins two lines of history without rewriting either of them.

A merge takes the work on one branch and brings it into another. Crucially, it does this non-destructively: every existing commit keeps its hash. Depending on the shape of the history, git does one of two things — it either slides a pointer forward (a fast-forward, no new commit at all) or it creates a single new commit with two parents (a three-way merge). Knowing which one happens, and why, is most of what there is to understand about merging.

1 · Fast-forward: when the branch hasn't diverged

You're on main at commit B. You branched feature off that same B and added C then D. Meanwhile main got no new commits — it is still at B. Since B is a direct ancestor of D, the two histories have not diverged: feature is simply ahead of main in a straight line. Step through git merge feature while on main:

Why no merge commit? A merge commit exists to reconcile two diverging lines. Here there is nothing to reconcile — every commit on feature already builds directly on top of where main is. So git takes the cheapest possible action: it just moves the main pointer forward to D. No new object is written; the result is a perfectly linear history. This is a fast-forward.

2 · --no-ff: forcing a merge commit on purpose

A fast-forward is tidy, but it makes the feature disappear as a unit — afterwards C and D look like they were always part of main. Many teams don't want that. git merge --no-ff feature tells git: even though you could fast-forward, don't — create a real merge commit anyway, so the feature shows up as a distinct bubble that can be reverted or read as one chunk. The merge commit M has two parents: the old main tip (B) and the feature tip (D). Reading git log --first-parent now skips C and D, summarising the branch as a single step — exactly why teams enable --no-ff for feature integration (and why GitHub's "Create a merge commit" button does the same thing).

ff vs no-ff at a glance. Plain git merge fast-forwards when it can (no divergence) and only builds a merge commit when it must. --no-ff always builds one. The opposite, --ff-only, refuses the merge unless a fast-forward is possible — handy in scripts that must keep history strictly linear.

3 · Three-way merge: when both branches moved

This is the case a merge commit was invented for. Starting from base B, main advanced to C while feature independently advanced to D then E. Now each branch has commits the other lacks — the histories have diverged, so a fast-forward is impossible. Step through git merge feature while on main:

It's called a three-way merge because git compares three snapshots: the merge base B (common ancestor), ours = C (the tip of the branch you're on), and theirs = E (the tip you're merging in). For each file it asks "did this change relative to B, and on which side?" — changes on only one side are taken automatically; changes on both sides to the same lines become a conflict (section 4).

Nothing is rewritten — contrast with rebase. C, D and E keep their original hashes; the only new object is the merge commit M, and M has two parents (C and E). The fork and the join are both preserved in history, a truthful record of what actually happened. Rebase, by comparison, would copy D and E onto C as new-hash commits D′ and E′ and erase the fork entirely.

4 · When git can't auto-combine: merge conflicts

The three-way merge auto-resolves any file changed on only one side. But when the same lines were edited differently on both sides, git has no way to know which version you want. It pauses the merge, writes both versions into the file surrounded by conflict markers, and waits for you to decide. A conflicted file looks like this:

<<<<<<< HEAD # "ours" — the branch you are on (e.g. main) port = 8080 ======= port = 3000 >>>>>>> feature # "theirs" — the branch being merged in

Everything between <<<<<<< HEAD and ======= is ours (your current branch); everything from ======= down to >>>>>>> feature is theirs. You edit the file by hand to the final text you want — keeping one side, the other, or a hand-merged blend — and delete all three marker lines. Then you finish the merge:

# 1. edit each conflicted file; remove the <<< === >>> markers # 2. stage the resolved files to tell git "this conflict is settled" git add config.txt # 3. create the merge commit (git pre-fills a merge message) git commit # git merge --continue does the same thing git merge --continue # changed your mind? throw the whole merge away, back to pre-merge state git merge --abort
"ours" and "theirs" are positional, not moral. During a normal merge, ours = the branch you ran git merge from (where HEAD is), theirs = the branch named in the command. git checkout --ours config.txt / --theirs config.txt grab one whole side fast. (Heads-up: during a rebase the two are swapped, because rebase replays your commits on top of the other branch.)

5 · Why merge is the safe choice for shared branches

This is the direct counterpart to rebase's golden rule. Rebase rewrites commits into new-hash copies, which is dangerous on history other people have already pulled. Merge never rewrites anything — existing commits keep their hashes, and integration only ever adds a merge commit on top. So merging is always safe on public, shared branches: nobody's existing commits move out from under them.

Rule of thumb. Rebase your local, unshared work to keep it clean and linear; merge to integrate anything that has been pushed and shared. Merge tells the truth about history (forks and joins included); rebase tells a tidier story at the cost of rewriting it.

6 · The default strategy, and --squash

Since Git 2.34 the default merge strategy is ort ("Ostensibly Recursive's Twin") — a faster, cleaner reimplementation of the old recursive strategy. It's what computes the three-way merge in section 3; you'll see Merge made by the 'ort' strategy in the output. You rarely need to think about it, but you can pick others with -s (e.g. -s octopus to merge several branches at once).

git merge --squash feature is a different animal entirely. It takes all of a branch's combined changes and stages them in your working tree as one set of changes — but it does not create a merge commit and does not record a second parent. You then run git commit yourself to produce a single ordinary commit. The feature's individual commits and the branch link are not carried over.

Left: a real --no-ff merge — commit M with two parents, feature history preserved. Right: --squash — a single new commit S with one parent containing the same file changes, but no link back to the feature branch. Use squash when you want the result but not the messy intermediate commits; use a real merge when the branch's history matters.

Check your understanding

You're on main, it has no commits since you branched, and you run git merge feature. Is a merge commit created?
No. Because main hasn't diverged — every feature commit builds directly on main's tip — git performs a fast-forward: it just slides the main pointer to the feature tip. No new commit object is written and history stays linear. To force a merge commit here you'd use git merge --no-ff feature.
After a three-way merge, do the commits C, D and E still have their original hashes?
Yes. Merge is non-destructive: it never rewrites existing commits. The only new object is the merge commit M, which records two parents (the tip of each side) and points at the merged tree. This is exactly why merge is safe on shared/public branches, and the key contrast with rebase, which replaces commits with new-hash copies.
In a conflict block, which side is "ours" and which is "theirs"?
Ours is the part between <<<<<<< HEAD and ======= — the branch you were on when you ran merge. Theirs is from ======= to >>>>>>> feature — the branch you named in the command. You edit to the final text, remove the markers, git add the file, then git commit (or git merge --continue). git merge --abort backs out entirely.
How does git merge --squash feature differ from a real merge?
A real merge creates a commit with two parents and preserves the branch's history. --squash stages all of the branch's changes as one set in your working tree but creates no merge commit and records no second parent; you then make a single ordinary commit yourself. The result is one flat commit with no link back to the feature branch — the individual feature commits are discarded.
