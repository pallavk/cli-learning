Understanding git rebase

Rebase does one thing: it replays your commits on top of a different base, creating brand-new commits.

The confusion around rebase melts away once you hold one idea: a commit is a snapshot plus a pointer to its parent. Rebase cannot "move" a commit, because a commit's identity (its hash) is built from its parent and contents. So rebase copies each of your commits onto a new parent, giving each a new hash, then slides your branch pointer to the last copy. The originals are left behind, unreferenced.

base history (e.g. main) your commits (about to be replayed) replayed copies (new hashes) orphaned originals (reflog only)

1 · Rebasing a branch onto another

You branched feature off main at commit B. Since then, a teammate added C to main. You want your work to sit on top of the latest main, as if you had started today. Step through git rebase main while on feature:

The Golden Rule of Rebasing. Never rebase commits that you have already pushed and others may have based work on. Rebase rewrites history (new hashes); if the old commits are public, everyone else still has them and your histories diverge. Rebase freely on your local, unshared branches; use merge for shared ones.

2 · Interactive rebase: cleaning up your own history

git rebase -i opens a todo list of the commits about to be replayed and lets you edit the plan: reorder lines, reword a message, squash several commits into one, edit to stop and amend, or drop to delete. Here you squash three messy "wip" commits into a single clean one with git rebase -i HEAD~3:

fixup vs squash. squash keeps both commit messages (you edit the combined one); fixup discards the squashed commit's message entirely. The workflow git commit --fixup=<sha> then git rebase -i --autosquash automates the todo-list ordering for you.

3 · Merge vs rebase — same goal, different history

Both integrate main's new work into feature. The difference is the shape of the history they leave behind.

Merge preserves exactly what happened (a fork and a join) and adds a merge commit with two parents. Rebase fabricates a tidy straight line, at the cost of rewriting your commits. Neither is "correct" — teams pick a convention. A common one: rebase your local feature to stay current, then merge --no-ff it into main so the feature stays visible as a unit.

4 · --onto: transplanting a range

The full form is git rebase --onto <newbase> <upstream> <branch>. It takes the commits that are on <branch> but not on <upstream>, and replays them onto <newbase>. It is the tool for "I branched off the wrong place" — e.g. you built topic on top of feature, but it should sit on main:

# move only topic's own commits onto main, dropping feature from underneath git rebase --onto main feature topic

5 · When a replay hits a conflict

Rebase applies your commits one at a time. If commit D doesn't apply cleanly onto the new base, rebase pauses and hands you the conflict. You are now mid-replay, not on a normal branch tip.

# 1. fix the conflicted files, then stage them git add <files> # 2. resume — do NOT run `git commit`, rebase owns the commit git rebase --continue # skip this commit entirely (rare) git rebase --skip # bail out completely, restoring the branch to its pre-rebase state git rebase --abort
You can always undo a rebase. Before it starts, git records where your branch was. git rebase --abort restores it mid-flight, and even after a "finished" rebase, git reflog shows the old tip so you can git reset --hard HEAD@{1} back. A rebase is never truly destructive until git's garbage collector runs (weeks later). See the reflog explainer.

Check your understanding

After git rebase main, do commits D and E still exist with the same hashes?
No. Their contents are replayed as new commits D′ and E′ with new hashes (because their parent changed, and a hash is computed from parent + tree + metadata). The original D and E still exist on disk but are now unreferenced — reachable only through the reflog until garbage-collected.
Why is rebasing a pushed/shared branch dangerous, but rebasing a local one fine?
Rebase replaces commits with new-hash copies. If teammates already pulled the originals, their branch still points at the old commits while yours points at the new ones — the histories diverge and a later push/pull creates duplicate commits and messy merges. On a branch only you have, nobody else holds the old commits, so replacing them is harmless.
You're mid-rebase resolving a conflict. Should you run git commit after git add?
No. During a rebase, git is the one creating each replayed commit. You stage the fix and run git rebase --continue; git then commits it (reusing the original message) and moves on to the next commit. Running git commit yourself would create an extra, unintended commit.
