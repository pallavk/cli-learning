Revert: undo by moving forward

A revert doesn't erase a commit. It writes a new commit whose changes are the exact inverse of an old one — cancelling its effect while leaving history fully intact.

There are two ways to "undo" in git, and they're opposites. git reset moves a branch pointer backwards, abandoning commits — it rewrites history and is unsafe on shared branches. git revert moves forwards, adding an anti-commit that subtracts the change — history grows, nothing is rewritten, and it's safe to push. This page is about the forward kind.

1 · Reverting a commit adds its inverse

History is A → B → C, and main (with HEAD) is at C. Commit B turned out to be wrong. You don't want to pretend B never happened — it's already pushed and others built C on top. Instead you cancel B with a new commit. Step through git revert B:

"Inverse diff" — literally. If B added three lines and deleted one, the revert commit ¬B deletes those three lines and re-adds the one. Applied on top of C, the net effect is "as if B's change is gone" — but B, and every commit after it, is still right there in the log. Git even pre-fills the message: Revert "B's subject".

2 · revert vs reset — the core distinction

This is the confusion worth burning in. Both "undo" B, but they leave the repository in very different shapes. Reset rips the pointer back to A and lets B and C fall off; revert leaves A → B → C standing and appends ¬B. Same goal, opposite mechanics:

The rule of thumb. If the commit you're undoing has left your machine — pushed, shared, or built on by others — use revert. If it's still private to you, reset is fine and tidier. Reverting is the only "undo" that never forces anyone else to rewrite their history.

3 · Reverting a merge commit

A merge commit has two parents, so "undo this merge" is ambiguous: inverse relative to which parent's line? You must say with -m:

# M is a merge of feature into main; main is parent 1, feature parent 2 git revert -m 1 M # undo the merge, keeping main's mainline → removes feature's changes git revert -m 2 M # the other mainline (rarely what you want)

-m 1 means "treat the first parent as the mainline, subtract everything the other side brought in". It works, but with a famous gotcha: git now thinks those feature changes are intentionally undone, so simply merging feature again later won't reintroduce them — you'd have to revert-the-revert first. That asymmetry is why many teams prefer to fix-forward rather than revert merges.

Undo an undo. Because a revert is just a commit, you can revert it: git revert <the-revert> re-applies the original change. And reverting multiple commits is one command — git revert C B or git revert A..C stacks an inverse commit for each (use --no-commit to fold them into one).
