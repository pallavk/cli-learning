Understanding git sparse-checkout

The branch still holds every file. Sparse-checkout just filters which ones get written to disk — and the switch is per-worktree, not per-repo.

Sparse-checkout tells git one thing: "only write a subset of the branch's files onto disk, ignore the rest." The branch's history is untouched — it still contains every file in every commit. Git simply filters which of those files materialise in your working directory. A file can be fully present in history yet invisible on disk because the filter excludes it. That isn't corruption; that's the feature working.

1 · The one-line idea: a filter in the middle

Think of three layers, with the sparse filter sitting between what git stores and what you see:

LayerHolds
Branch history (in .git)every file, all commits — complete and untouched
Sparse filtera list of path patterns to keep
Working directory (on disk)only the files matching the filter

The one sentence to remember. Sparse-checkout is a checkout-level filter, never a history-level edit. Your commits keep every file; only the set of files written into the working directory shrinks. Turn the filter off and the missing files reappear instantly — nothing was ever deleted.

2 · Is it repo-level? No — it's per-worktree

This is the subtlety that trips people up. Both the on/off switch (core.sparseCheckout) and the pattern list can live in a single worktree's private config, not the shared repo config. That is why one worktree of a repo can be filtered down to a sliver while another worktree of the same repo sees the full tree.

Config scopeWhere it livesShared by
System/etc/gitconfigevery repo on the machine
Global (user)~/.gitconfigall your repos
Repo (local).git/configall worktrees of this repo
Worktree.git/worktrees/<name>/config.worktreejust that one worktree

Sparse-checkout uses the bottom row. Unlocking that per-worktree layer requires turning on a repo extension:

# let each worktree carry its own config (needed for per-worktree sparse) git config extensions.worktreeConfig true

Without worktrees in play, core.sparseCheckout simply sits in .git/config and filters that repo's single working copy. Either way the rule holds: it's always "checkout level," never "history level" — your commits are never touched.

Why it has to be per-worktree. Worktrees already share one object store and one set of refs (see git worktree). The only things that are private to each worktree are its files on disk, its index, and its HEAD. "Which files to write to disk" is exactly that kind of per-checkout decision — so the sparse filter belongs in the worktree's own config.worktree, right alongside the rest of its private state.

3 · A real example: the beads-sync worktree

In one repo, a linked worktree called beads-sync exists for a single purpose — to commit the .beads/ state folder. So it's filtered down to just that one path, while the main worktree keeps the whole tree:

# inside .git/worktrees/beads-sync/ config.worktree → [core] sparseCheckout = true info/sparse-checkout → /.beads/ # result on disk in that worktree $ ls -A .beads .git # nothing else from the branch

The proof that it's per-worktree is that the same command gives a different answer depending on which worktree you run it from:

# from the MAIN worktree (not sparse) $ git sparse-checkout list fatal: this worktree is not sparse # from the beads-sync worktree (sparse) $ git sparse-checkout list /.beads/

Same repo, same shared history, two different answers — because the filter lives in each worktree's own config, not the repo's.

4 · The gotcha: a missing file that isn't missing

When a tool tries to write a file outside the filtered paths in a sparse worktree, it looks like the write failed or the file vanished. Step 3 of the diagram above shows exactly this: an attempt to write src/app.js in a worktree filtered to /.beads/. The file isn't missing — the sparse filter is excluding that path on purpose.

Don't force it as a first move. Reaching for git checkout --ignore-skip-worktree-bits or reapplying patterns is the wrong reflex. First read the filter to understand why the path is excluded — the exclusion is usually intentional.

The two commands that tell you what's going on:

git config core.sparseCheckout # true / false — is the filter on? git sparse-checkout list # show the active patterns

The fix is almost always to work inside the allowed paths, or to do the work in the correct (non-sparse) worktree instead — not to fight the filter.

Check your understanding

A file is in the branch's history but not on disk in your worktree. Is the repo corrupt?
No — that's sparse-checkout working as designed. History in .git still holds every file; the sparse filter is just excluding that path from the working directory. Turn the filter off (or add the path to the pattern list) and the file reappears immediately. Nothing was deleted.
You run git sparse-checkout list in two worktrees of the same repo and get different output. Bug?
No. Sparse-checkout is configured per-worktree — the switch and pattern list live in each worktree's own .git/worktrees/<name>/config.worktree, not the shared .git/config. One worktree can be filtered (returns its patterns) while another isn't sparse at all (returns fatal: this worktree is not sparse). Same repo, same history, different filter.
A tool reports it can't write src/app.js in a worktree filtered to /.beads/. What's the right first step?
Diagnose before forcing. Run git config core.sparseCheckout (is the filter on?) and git sparse-checkout list (what does it keep?). You'll see the path is excluded on purpose. The fix is to work inside the allowed paths, or switch to the correct non-sparse worktree — not --ignore-skip-worktree-bits.
Does sparse-checkout change your commits or branch history?
Never. It's strictly checkout-level: it decides which files get written to the working directory, nothing more. Commits, trees, and blobs in the object store are untouched, and so is every other worktree that doesn't share the filter. It's a view, not an edit.
Related: sparse-checkout makes the most sense once worktrees click — see git worktree for the shared-object-store model that explains why the filter is per-worktree.
