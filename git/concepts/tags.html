Tags: permanent names for a commit

A branch is a pointer that moves as you commit. A tag is a pointer that doesn't — you nail it to one commit and it stays there forever. That single difference is what makes tags the right tool for marking releases.

When you ship v1.0.0, you want a name that will always mean exactly the bytes you released — not "wherever main happens to be now." Tags give you that fixed handle. There are two kinds (lightweight and annotated), they don't travel with a normal push, and checking one out lands you in detached HEAD.

1 · Lightweight vs annotated

A lightweight tag is just a name pointing at a commit — like a branch that never moves. An annotated tag is a real object in git's database with its own author, date, message, and optional GPG signature. Use annotated for anything you publish.

git tag v1.0.0 # lightweight: just a name → commit git tag -a v1.0.0 -m "Release 1.0" # annotated: object w/ tagger, date, message git tag -s v1.0.0 -m "Release 1.0" # signed (GPG) annotated tag git tag v0.9 9fceb02 # tag an OLD commit by hash, after the fact git show v1.0.0 # annotated tags carry their own metadata
AspectLightweightAnnotated
Stored asA ref pointing at a commit (no object)A full tag object in the database

2 · A tag doesn't move

This is the whole idea. Tag a commit, then keep working and watch main march on while the tag stays exactly where you put it.

Branch = verb, tag = noun. A branch answers "where is work happening?" and moves with every commit. A tag answers "what did we call this exact point?" and never moves on its own. That's why you build on branches but reference tags.

3 · Tags don't ride along with push

A plain git push sends commits and branch updates — not tags. You push tags explicitly, which surprises people the first time a teammate can't see v1.0.0.

git push origin v1.0.0 # push one specific tag git push origin --tags # push all local tags git tag -d v1.0.0 # delete the tag locally git push origin --delete v1.0.0 # delete it on the remote too

4 · Checking out a tag = detached HEAD

A tag isn't a branch, so there's nothing for HEAD to follow. Check one out and you land in detached HEAD — fine for inspecting or building a release, but make a branch before you commit, or your work will be hard to find.

Releases build on tags. Semantic versioning names tags v<major>.<minor>.<patch> (e.g. v2.4.1). Most forges turn an annotated tag into a "Release" page, and CI systems commonly trigger a publish/deploy when you push a tag matching v*. The tag is the anchor the whole release process hangs off.
