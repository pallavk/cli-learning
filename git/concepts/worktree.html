Understanding git worktree

One repository, many working directories — each on a different branch, all sharing a single history.

A normal clone is a one-to-one deal: one folder of files, tied to one checked-out branch. To work on a second branch you switch in place — which means stashing or committing whatever you were doing first. git worktree breaks that one-to-one rule. It lets a single repository project several working directories at once, each parked on its own branch, in its own folder — without a second clone and without duplicating any history.

1 · The core mental model

Every git repository has two halves that we usually see as one folder:

• The object store & refs — the .git directory: all commits, trees, blobs, branch pointers, tags, the stash, config and remotes. This is the repository's brain, and there is exactly one of it.
• A working directory — the actual files on disk for one branch, plus an index (staging area). This is just a view the brain projects.

A clone glues exactly one working directory to that brain. git worktree lets the same brain drive multiple working directories simultaneously. They all read and write the same commits, branches and stash — they just each show a different branch's files. Step through adding one:

.git object store
~/proj/.git
all commits branches + tags stash · config · remotes
~/projfeature
main working directory · half-done feature work
.git/  (real directory — the brain lives here)

▲ linked via .git file

~/proj-hotfixmain
linked worktree · urgent bug fix on a clean tree
.git  → gitdir: ~/proj/.git/worktrees/proj-hotfix
(no command yet)
The one sentence to remember. A worktree is a second (or third) checkout of the same repository, in a different folder, on a different branch — sharing one object store and one set of refs. No second clone. No duplicated objects. No stashing to switch.

2 · Why this matters

The everyday problem worktrees solve: you're deep in feature, the tree is dirty and half-broken, and something urgent lands on main. Below, the same A→B→C history is shared by every worktree — each branch pointer just selects a different commit to materialise on disk. The worktrees don't copy history; they point into it.

Concrete wins: fix an urgent bug on main in ../hotfix while your feature stays exactly as you left it · run a long build or test suite on one branch while you keep coding in another · review a PR branch in a separate folder your editor can open side-by-side · diff two branches in two editor windows instead of flipping one window back and forth.

3 · The commands

git worktree add <path> <branch> is the workhorse. It creates a new directory at <path>, checks <branch> out there, and writes a lightweight link back to the main repo — no copying of .git.

# check out an EXISTING branch into a sibling folder git worktree add ../hotfix main # create a NEW branch and check it out in one step git worktree add -b reviewpr ../reviewpr origin/pr-42 # see every worktree, its path, HEAD commit and branch git worktree list # detach a worktree cleanly (removes the folder + admin link) git worktree remove ../hotfix # clean up admin entries for folders that were deleted by hand git worktree prune

git worktree list prints one line per worktree — the main one first, then each linked one with its absolute path, short HEAD hash, and [branch]. remove is the polite way to delete: it refuses if the worktree has uncommitted changes (use --force to override), then deletes both the folder and the internal bookkeeping.

The link is a file, not a folder. In a linked worktree, .git is a tiny text file containing one line — gitdir: /path/to/main/.git/worktrees/<name> — pointing back at the real object store. That per-worktree admin folder holds only what is genuinely per-checkout: that worktree's own HEAD, index, and logs. Commits, branches, stash and config still live once, in the main .git.

4 · The key constraint: a branch can't be checked out twice

This is the rule that surprises everyone. Because every worktree shares the same branch pointers, git refuses to check out a branch that is already checked out in another worktree. If it allowed two folders to sit on main, a commit in one would silently yank the other's HEAD out from under it. So git stops you up front:

.git object store
~/proj/.git
main → C one pointer, shared
~/projmain
already has main checked out

▼ tries to claim the same branch

~/proj-2main ✗
git worktree add ../proj-2 main
(no command yet)
The fix is almost always a different branch. You rarely want the same branch twice. Use git worktree add -b hotfix ../hotfix main to branch a fresh hotfix off main and check that out instead. If you genuinely must (e.g. a build that won't touch HEAD), --force overrides the guard — but then you own the risk of two trees fighting over one ref.

What worktrees share — and what's per-tree

Everything in the object store is shared, so changes are instantly visible everywhere: commit in one worktree and the new commit, and any branch you moved, is immediately seen by the others. The stash, config, remotes and tags are all single, shared copies. What's private to each worktree is only its working tree files, its index/staging area, and its own HEAD (which branch it's on).

5 · Worktree vs. the alternatives

Worktrees sit in the gap between two things you already know. Here is the same "I need to touch another branch" situation solved three ways:

git stash (left) keeps you in one folder: you shelve your dirty changes, switch branch in place, deal with the urgent thing, switch back, and stash pop. It works, but it's context-switch friction — your editor reloads, your build cache churns, and a forgotten stash bites later. A second git clone gives you a separate folder but duplicates the entire object store on disk and gives you separate branches, stash, config and remotes that drift apart. Worktrees (right) are the middle ground: a separate folder like a clone, but one shared brain like a stash-free switch — no shelving, no disk bloat, no drift.

Deleting a worktree by hand leaves a stale entry. If you rm -rf ../hotfix instead of git worktree remove, the main repo still lists a worktree that no longer exists. That's harmless but noisy — and it keeps the branch "claimed", so you can't check it out elsewhere. git worktree prune sweeps away those orphaned admin entries and frees the branch again.

Check your understanding

Does git worktree add ../x feature copy the repository's history into ../x?
No. It copies no history at all. The new folder gets the working files for feature plus a one-line .git file pointing back to the original .git object store. All commits, branches, tags, the stash and config stay in the single main .git and are shared. That's the whole point — unlike a second clone, nothing on disk is duplicated.
Why does git refuse git worktree add ../two main when main is already checked out?
Because all worktrees share one set of branch pointers. If two folders both sat on main, committing in one would advance main and silently move the other folder's HEAD, leaving its files out of sync with the ref it thinks it's on. Git blocks the second checkout to prevent that conflict. Check out a different branch, or create a new one with -b.
You commit on a branch in worktree A. Is it visible in worktree B?
Yes, immediately — if B is on a branch that includes it. There is one object store and one set of refs, so a new commit, a moved branch pointer, a new tag, or a stash entry created anywhere is instantly part of every worktree's shared brain. Only the working files, the index, and which branch each worktree has checked out (its HEAD) are per-worktree.
You ran rm -rf ../hotfix to delete a worktree. What's left, and how do you clean it?
The folder is gone, but the main repo still keeps an admin entry under .git/worktrees/ that lists the now-missing worktree (you'll still see it in git worktree list), and it keeps that branch marked as checked-out so you can't claim it elsewhere. Run git worktree prune to drop the stale entry and free the branch. Next time, prefer git worktree remove ../hotfix, which does both cleanly.
