git reflog: your safety net

The reflog is the reason git is undo-able. Almost nothing you do is truly lost — and this is how you get it back.

Every time HEAD moves — a commit, a checkout, a reset, a rebase, a merge — git quietly appends a line to a log recording where HEAD just was. That log is the reflog. It means that even when a command appears to throw work away, git still remembers the exact hash it was sitting at. Learn to read the reflog and the fear drains out of git: a bad reset --hard, a botched rebase, a deleted branch, an amended-away commit — all recoverable.

The reflog is yours alone. It lives in .git/logs/ and is local only — it is never pushed, fetched, or cloned. Your teammate's reflog is theirs; yours is yours. A fresh clone has an empty reflog. So the reflog protects your own local missteps, not shared history.

1 · Where do "lost" commits go?

The crucial mental model: a destructive-looking command does not delete commits. A commit is an object in git's database, addressed by its hash. When you reset --hard past it, delete the branch pointing at it, or rebase it away, you only remove the references that point to it. The object itself stays on disk, now unreferenced — and the reflog still records the hash. That is your thread back to it.

Garbage collection is slow on purpose. Unreferenced objects survive until git gc prunes them — and the defaults are generous: ~90 days for objects still reachable from a reflog entry (gc.reflogExpire), and ~30 days for fully unreachable ones (gc.reflogExpireUnreachable). You almost always have weeks to recover. The reflog is what keeps those entries "reachable," buying you the longer window.

2 · Reading the reflog

Plain git reflog is shorthand for git reflog show HEAD. It prints one line per HEAD movement, newest first:

6fac19d HEAD@{0}: commit: commit D 4036700 HEAD@{1}: commit: commit C cf529a5 HEAD@{2}: commit: commit B d79166f HEAD@{3}: commit (initial): commit A

Each line is: abbreviated hash · the selector · the operation · description. The selector HEAD@{n} means "where HEAD pointed n moves ago". HEAD@{0} is right now; HEAD@{1} is one HEAD-move in the past, and so on. You can use that selector anywhere git wants a commit: git show HEAD@{2}, git diff HEAD@{5}, git reset --hard HEAD@{1}.

HEAD@{n} is NOT HEAD~n. This trips everyone up. HEAD~2 walks two parents back in ancestry — it is a structural fact about the commit graph, the same for everyone, forever. HEAD@{2} walks two entries back in your reflog — a time-travel selector through your local history of moves, and it changes every time you do anything. After a reset --hard, the commit you abandoned is not at HEAD~1 (it is no longer an ancestor) but it is at HEAD@{1} (HEAD was just there). Curly braces = reflog/time; tilde = ancestry.

Other ways to query it

# the reflog of a specific branch, not HEAD git reflog show main # time-based selectors — git interpolates from reflog timestamps git reflog show main@{yesterday} git show HEAD@{2.hours.ago} git diff main@{one.week.ago} main

Branch reflogs (main@{n}) record where that branch tip has been; HEAD's reflog records where you've been. Time selectors like @{yesterday} are resolved against the recorded timestamps, so they only reach as far back as the reflog itself.

3 · Rescue story: undoing a bad reset --hard

This is the canonical save. You have A→B→C→D on main, HEAD at D. You meant to unstage something but typed git reset --hard B. The branch jumps back to B and C, D vanish from your working tree. Step through the disaster — and the one-line recovery:

Why HEAD@{1} works here. The reset itself is a HEAD movement, so it became the new HEAD@{0}. That pushed the moment before the reset — when HEAD was still at D — to HEAD@{1}. Resetting hard to HEAD@{1} lands the branch right back on D. You could equally use D's hash from the reflog: git reset --hard 6fac19d.

4 · Rescue story: recovering a deleted branch

You finished a spike on a feature branch, switched back to main, and ran git branch -D feature to tidy up — then realized you still needed that commit. Deleting a branch only removes the pointer; the commit is still in the object database, and HEAD's reflog recorded the hash back when you were on the branch.

Finding the hash. git branch -D conveniently prints Deleted branch feature (was cad82be) — that's the hash. If you missed it, git reflog shows the checkout: moving from feature to main line; the commit just above it is the branch's last tip. Then git branch feature cad82be resurrects it. (git checkout cad82be followed by git switch -c feature works too.)

5 · More recovery recipes

The pattern is always the same: find the hash in the reflog, then point something at it.

Commits orphaned by a rebase

A rebase replays your commits as new-hash copies and leaves the originals unreferenced (see the rebase explainer). If a rebase went wrong — bad conflict resolution, dropped the wrong line — the pre-rebase tip is in the reflog:

# find the entry just before the rebase started git reflog # a1b2c3d HEAD@{0}: rebase (finish): returning to refs/heads/feature # 9f8e7d6 HEAD@{5}: rebase (start): checkout main ← HEAD was here before # option A: move the branch back to the pre-rebase tip git reset --hard HEAD@{5} # option B: inspect first, then branch off the good hash git checkout 9f8e7d6 git switch -c feature-recovered

A commit you --amended away

git commit --amend does not edit a commit in place — it creates a replacement commit and moves the branch to it. The original (with the old message or the code you amended out) is orphaned but logged:

5574632 HEAD@{0}: commit (amend): amended message 0049d32 HEAD@{1}: commit (initial): original message ← the pre-amend commit # get the original content back git show HEAD@{1} git reset --hard HEAD@{1} # or cherry-pick / branch off the hash
Lost a commit that's truly orphaned? If even the reflog entry is gone, git fsck --lost-found scans the object database for dangling commits — unreferenced objects gc hasn't pruned yet — and lists their hashes for you to inspect with git show. It's the last-resort net under the reflog net.

6 · How it relates to reset and rebase

The reflog is the undo button behind both of git's "scary" history commands. reset moves a branch pointer (see the reset explainer) and rebase rewrites a series of commits (see the rebase explainer) — and both record where HEAD was before they ran, so HEAD@{1} (or the hash) walks you straight back. Knowing the reflog has your back is exactly what lets you use reset --hard and rebase without flinching.

Check your understanding

After git reset --hard HEAD~3, where is the commit you were just on, and how do you get back?
It is orphaned but intact in the object database, and the reset itself was logged. Because the reset is the new HEAD@{0}, the commit you were on is now HEAD@{1}. Recover with git reset --hard HEAD@{1} (or use the hash shown in git reflog). Note it is not at HEAD~1 — after the reset it is no longer an ancestor of HEAD.
What's the difference between HEAD@{2} and HEAD~2?
HEAD~2 is ancestry: two parent-hops back in the commit graph — the same for everyone, an immutable fact about the history. HEAD@{2} is the reflog: two HEAD-moves back in your local history of operations — it changes every time you do anything, and only exists on your machine. Tilde = graph/structure; curly braces = time/your moves.
You cloned a teammate's repo to recover a commit they lost. Will their reflog help?
No. The reflog lives in .git/logs/ and is never transferred by push, fetch, or clone. Your fresh clone has only its own (essentially empty) reflog. Only their local repo holds the reflog entry that points at their lost commit — they have to run the recovery themselves.
Is a "lost" commit gone immediately after a destructive command?
No. The command only removed the references pointing to it; the commit object stays on disk, unreferenced. Git's garbage collector won't prune it until expiry — roughly 90 days while a reflog entry still references it, ~30 days once fully unreachable. Until then git reflog (or git fsck --lost-found) can find it.
