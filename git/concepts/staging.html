The staging area: the commit you're composing

Between your edited files and the saved history sits a third thing: the index, a.k.a. the staging area. It's not a list of filenames — it's a complete, editable draft of your next commit. Once you see it that way, add, restore, and the two columns of git status stop being mysterious.

Most version-control tools commit whatever's changed. Git makes you assemble the commit first, in the index, with git add. That extra step is a feature: it lets you commit the one logical change you mean to — not the debug print you left three lines above it.

The three places your changes live

working tree ── git add ──▶ index (staging) ── git commit ──▶ HEAD your edits the next commit history ▲ ▲ └── git restore <f> ──┘ git restore --staged <f> (discard edits) (unstage, keep edits)

Every file is in one of these states, and add / commit / restore are just the moves between them.

1 · The index is a real, editable draft

The index starts out matching HEAD — an empty draft. As you add, you copy changes into it, building up the commit. commit then freezes whatever the index holds into history.

The index holds content, not a to-do list. git add file doesn't mark the file "to be committed later" — it copies the file's current bytes into the index right then. Edit the file again afterwards and that newer change is not staged; you'll have the same file in both columns (see below). This trips up everyone once.

2 · git status has two columns

Short status shows two character slots per file: the left is the index (staged), the right is the working tree (unstaged). A file can appear in both at once.

# git status -s (left = staged · right = unstaged) M  app.py # staged change, nothing unstaged  M utils.py # modified, but NOT staged MM config.py # staged AND further unstaged edits — same file, both columns A  new.py # newly added to the index ?? notes.md # untracked — git isn't watching it yet

3 · Stage part of a file

The real payoff of a separate staging step: you can stage individual hunks, not just whole files. Commit the fix, leave the debugging behind — in the same file.

git add -p # walk each hunk: y stage · n skip · s split · q quit # stage the real change, leave the stray print() unstaged git diff # what's still UNstaged (working tree vs index) git diff --staged # what you're ABOUT to commit (index vs HEAD)
Two diffs, because there are two gaps. git diff compares the working tree to the index — "what haven't I staged yet?" git diff --staged compares the index to HEAD — "what will this commit actually contain?" Reaching for the wrong one is why a diff sometimes looks empty right after you stage.

4 · Undoing, at each stage

GoalCommandNote
Unstage a file (keep the edits)git restore --staged <f>older form: git reset HEAD <f>
Discard working-tree editsgit restore <f>destructive — the edits are gone
Unstage everythinggit restore --staged .index back to matching HEAD
Stage everything (incl. deletions)git add -Awhole working tree → index
Commit, skipping the indexgit commit -atracked files only; no -p granularity
commit -a is a shortcut, not a synonym. It auto-stages modifications to already-tracked files, then commits — handy, but it skips the whole point of the index: you lose hunk-level control and it won't pick up new untracked files. When the commit matters, stage deliberately.
