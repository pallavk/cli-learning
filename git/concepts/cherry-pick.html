Cherry-pick: copying one commit

A cherry-pick doesn't move a commit and doesn't merge a branch. It copies the changes of one specific commit and lays them down as a brand-new commit on your current branch.

Every commit records a diff — the change it introduced relative to its parent. git cherry-pick X takes that diff and replays it on top of wherever you are now, producing a fresh commit with the same change and message but a different hash. The original X never moves. Think of it as copy-and-paste for a single commit's work.

1 · One commit, copied onto your branch

You're on main at B. A colleague's feature branch added C, D, E — but you only need the bug-fix in D, not C or E. You can't merge (that would drag in everything) and you can't rebase (you don't want feature's whole line). You want exactly D's change. Step through git cherry-pick D while on main:

Why the hash changes. A commit's hash is computed from its content and its parent, author, and timestamp. The copy D' has a new parent (B instead of C) and a new commit time, so it hashes differently — even though the file changes and the commit message are identical. That is why cherry-pick is a copy, not a move: D still sits on feature, unchanged.

2 · Picking several commits

You can pick more than one. git cherry-pick D E replays them in order, stacking a new commit for each onto your branch. (A range works too: git cherry-pick C..E picks everything after C up to E — note the start of the range is excluded.)

The duplicate-commit trap. If you cherry-pick D onto main and later merge feature into main, the same change exists on two different commits (D and D'). Git is usually smart enough to notice they introduce identical patches and not apply it twice — but if anything nearby shifted, you can get a confusing conflict over a change you "already have". This is the classic downside of cherry-picking across branches that will later be merged.

3 · Conflicts, and leaving a trail

Replaying a diff onto a different base can fail to apply cleanly — a cherry-pick conflict, resolved exactly like a merge conflict:

# after a conflicting pick: git cherry-pick D # → CONFLICT (content): Merge conflict in app.py # fix the files, then: git add app.py git cherry-pick --continue # finish the copy # or bail out entirely: git cherry-pick --abort # back to before the pick git cherry-pick --skip # drop just this one, continue a range

Two flags worth knowing. -x appends a line (cherry picked from commit <hash>) to the new commit's message — invaluable for tracing where a backported fix came from. -n (--no-commit) applies the change to your working tree and index but doesn't commit, so you can combine several picks into one commit or tweak before committing.

cherry-pick vs rebase vs merge. They're three answers to "I want work from over there".
· merge — bring in a whole branch and record that you did, with a merge commit.
· rebase — move your entire series of commits onto a new base (it's really cherry-pick applied to a run of commits, then the branch moved).
· cherry-pick — surgically copy one or a few chosen commits, leaving both branches otherwise as they were. The canonical use: backporting a single hotfix from main onto a release-1.x branch.
Inspect it yourself. git cherry main feature shows which commits on feature are not yet on main (by patch, so already-picked ones are marked) · git log --oneline on both branches reveals the twin commits with different hashes · git show <hash> confirms the diffs match.
