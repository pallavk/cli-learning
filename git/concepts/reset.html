The Three Trees & git reset

Every confusing thing git reset does becomes obvious once you can see the three "trees" it moves between: HEAD, the index, and your working directory.

Git is not just a pile of commits. At any moment it is juggling three trees — three snapshots of your project. git add, git commit, git checkout and git reset are all just different ways of copying one tree onto another. Learn the three trees and reset's three modes (--soft, --mixed, --hard) stop being scary.

1 · The three trees

A "tree" here just means a full snapshot of your files. Git keeps three of them in play:

HEAD
The last commit — the snapshot your next commit's parent will be.
v2
Index (staging area)
The proposed next snapshot — what add writes to, what commit freezes.
v2
Working Directory
Your actual files on disk — what you edit in your editor.
v3

Here you've edited a file to v3 on disk but not staged it: HEAD and the index still hold v2. That gap — git status calls it "changes not staged for commit" — is the difference between two of the trees.

Working Dir ──git add──▶ Index ──git commit──▶ HEAD (new commit)

git add copies the working directory into the index. git commit takes whatever is in the index, freezes it into a new commit, and advances HEAD (and the current branch) to point at it. Everything flows rightward when you build history. checkout/switch and reset are the commands that push contents the other way — from a commit back into the index and working directory.

Why "the index" matters. Because the index sits between your files and your commits, you get to decide exactly what goes into each commit — staging some files (or even some hunks, via git add -p) and leaving others. git reset's whole job is to reposition these three trees relative to each other.

2 · What git reset <commit> actually does

The commit-moving form of reset is a two-stage operation, and the mode flag decides how far down the stack it goes:

# 1. ALWAYS: move the current branch pointer (and HEAD) to <commit> # 2. --soft : stop here. Index + working dir untouched. # 3. --mixed : also reset the INDEX to match <commit> (this is the DEFAULT) # 4. --hard : also reset the WORKING DIR to match <commit> (discards edits) git reset [--soft | --mixed | --hard] <commit>

Each mode is a strict superset of the one above it. --soft touches one tree (HEAD), --mixed touches two (HEAD + index), --hard touches all three (HEAD + index + working dir). The legend below colours each step's state.

3 · git reset --soft HEAD~1

Start from A→B→C, branch main and HEAD at C. --soft moves the branch back to B and stops. C's changes are left sitting in the index, fully staged — as if you'd run git add but not yet git commit. This is the clean way to "undo the last commit but keep its work ready to re-commit".

4 · git reset --mixed HEAD~1 (the default)

Same branch move to B, but now git also resets the index to match B. C's changes survive in your working directory but are no longer staged — git status shows them as "changes not staged for commit". Running git reset HEAD~1 with no flag does exactly this. It's the everyday "uncommit and unstage, let me redo the staging" move.

5 · git reset --hard HEAD~1 ⚠ destructive

Branch moves to B, the index resets to B, and the working directory is overwritten to match B. C's changes are gone from all three trees. The commit C itself still lingers in the object database (recoverable via reflog for a while), but any uncommitted edits you had in the working tree are destroyed with no reflog to save them.

What reflog can and cannot recover. git reset --hard on committed work is recoverable: git reflog still lists the old tip, so git reset --hard HEAD@{1} brings it back (until garbage collection, typically weeks). But --hard also blows away any uncommitted edits in the working directory, and those were never in a commit — the reflog cannot help. Commit or git stash before a hard reset if you're unsure.

6 · Side-by-side: where do the changes end up?

All three start from the same A→B→C with HEAD at C and run … HEAD~1. The only difference is the final state of the index and working directory:

ModeBranch / HEADIndexWorking DirC's changes end up…
--softmoved to Buntouched (still C)untouched (still C)staged, ready to commit
--mixed (default)moved to Breset to Buntouched (still C)unstaged in working dir
--hardmoved to Breset to Breset to Bdiscarded (gone)

Read it as a ladder: each mode does everything the one above it does, plus resets one more tree. The branch pointer always moves; the question is only how many of the other two trees follow it back.

7 · reset vs revert vs restore

Three commands sound like "undo" but do very different things. Pick by asking: am I rewriting history, recording an inverse, or just touching files?

CommandMoves HEAD / branch?What it doesSafe on shared history?
git resetYesMoves the branch pointer back; optionally resets index/working dir. Rewrites history.No — never on pushed commits
git revertNo (adds a commit)Creates a new commit that applies the inverse of a target commit. History stays intact.Yes — the safe undo
git restoreNoOperates on files only. Copies content from a commit/index into the index or working dir.Yes — touches no commits

reset vs revert on a shared branch

If C has been pushed and a teammate already based work on it, reset --hard HEAD~1 rewrites the branch out from under them — their history and yours diverge. git revert C instead adds a new commit C′ that undoes C's changes, leaving every existing hash in place. Same end state of the files, totally different (and safe) history:

Left: reset discards C and rewinds the pointer — fine locally, dangerous if C was shared. Right: revert keeps C and appends C′ (an "anti-C") — the history of what happened is preserved, which is exactly what collaborators need.

The restore family — files, never HEAD

git restore is the modern, file-scoped sibling. It never moves a branch:

# unstage a file — same effect as a mixed-reset of ONE path git restore --staged <file> # (older form: git reset HEAD <file>) # discard working-dir changes to a file, reverting it to the index/HEAD version git restore <file> # ⚠ throws away your edits to that file # do both: discard staged AND working-dir changes git restore --staged --worktree <file>

git reset <paths> — the path form moves nothing

One more source of confusion: git reset with file paths behaves completely differently from git reset <commit>. With paths, git does not move HEAD at all — it only copies those paths from HEAD (or a given commit) back into the index, i.e. it unstages them. There is no --soft/--hard with paths.

# unstage these files — HEAD stays put, working dir untouched git reset HEAD <file>... # vs. the commit form, which MOVES the branch pointer git reset --mixed HEAD~1
Two reset commands wearing one name. git reset <commit> moves your branch (history surgery). git reset <paths> just unstages files (no pointer moves). The default mode of the commit form is --mixed; the path form is effectively always a per-file mixed reset. When in doubt, prefer the clearer modern spellings: git switch/git restore --staged for the file cases, git reset --soft|--mixed|--hard only when you truly mean to move the branch.

Check your understanding

You ran git reset --soft HEAD~1. Where are the last commit's changes now, and what one command re-creates the commit?
The branch moved back one commit, but the index and working directory were untouched — so the changes are still fully staged. A plain git commit (which freezes the index) re-creates a commit from exactly those changes. This is the idiomatic way to "redo the last commit message" or fold the last commit into a re-stage.
Which reset mode is the default, and what state does it leave your changes in?
--mixed is the default (so git reset HEAD~1 = git reset --mixed HEAD~1). It moves the branch and resets the index to match, but leaves the working directory alone — so the undone commit's changes are present on disk but unstaged. git status shows them under "changes not staged for commit".
You did git reset --hard HEAD~1 and lost work. The commit was pushed earlier; you'd also made uncommitted edits. What's recoverable?
The committed work is recoverable: git reflog still shows the old tip, so git reset --hard HEAD@{1} (or the commit's hash) restores it. The uncommitted edits in your working directory are not — they were never in any commit, so the reflog has nothing to point to. Hard reset overwrote them. Lesson: commit or git stash before a hard reset.
Commit C is already pushed and a colleague branched off it. You need to undo C's changes. reset or revert?
git revert C. It creates a new commit that inverts C's changes while leaving C — and every other hash — in place, so your colleague's history still lines up. reset would rewrite the branch and make your histories diverge, forcing a dangerous force-push and breaking everyone downstream. Reset is for local, unshared history; revert is the shared-history-safe undo.
What's the difference between git reset HEAD file.txt and git reset HEAD~1?
The first has a path, so it's the path form: it unstages file.txt and moves nothing — HEAD and the working directory are unchanged (equivalent to git restore --staged file.txt). The second has no path, so it's the commit form: it moves the branch back one commit (a mixed reset by default). Same verb, two very different operations.
