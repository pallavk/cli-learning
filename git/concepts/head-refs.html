HEAD, refs & the detached HEAD

Branches, tags, and HEAD are all just pointers — tiny files holding a commit hash. Once you see the pointers, the rest of git stops being magic.

Git's history is a graph of immutable commits, each naming its parent. Everything you do — switch branches, commit, tag a release — is really just moving little named pointers around that graph. A branch is a movable pointer. A tag is a pinned pointer. HEAD is the pointer that says "you are here". The dreaded detached HEAD is simply HEAD pointing straight at a commit instead of at a branch.

First — what is a ref?

All of those pointers share one underlying type: the ref (short for reference). A ref is simply a name that points to a commit, stored as a tiny file under .git/refs/ whose contents are a commit hash. That's the whole idea — and almost every git "thing" you can name is just a ref living in a different folder:

# a peek inside a repo's .git directory .git/ ├─ HEAD # → ref: refs/heads/main (a symbolic ref — points at another ref) ├─ refs/heads/main # → 4399030… # a branch (moves when you commit) ├─ refs/heads/feature # → a1b2c3d… ├─ refs/tags/v1.0 # → 9f8e7d6… # a tag (pinned, never moves) ├─ refs/remotes/origin/main # → 4399030… # remote-tracking (last fetch) └─ packed-refs # many refs packed into one file for efficiency

The folder a ref lives in is what gives it its behaviour:

Kind of refLives underWhat makes it special
Branchrefs/heads/A movable pointer. Whenever you commit while it's the current branch, git rewrites its file to the new commit — it follows you forward.
Tagrefs/tags/A pinned pointer, meant never to move (a release marker). Lightweight = points at a commit; annotated = points at a tag object holding a message, tagger and date.
Remote-trackingrefs/remotes/A read-only local snapshot of where a remote's branch was at your last git fetch (e.g. origin/main). You never commit on it; fetch moves it.
HEAD.git/HEADThe special "you are here" ref. Normally symbolic (ref: refs/heads/<branch>); detached when it holds a commit hash directly.
Scratch refs.git/ top levelORIG_HEAD, FETCH_HEAD, MERGE_HEAD — temporary pointers git writes during operations so you (and it) can refer back to them.

Two refinements you'll meet everywhere:

Symbolic refs. A ref can point at another ref instead of a commit — its file literally contains ref: refs/heads/main. HEAD is the everyday example: it points at a branch, which points at a commit. That one extra hop is exactly why a single git commit advances both your branch and HEAD — HEAD never changed, the branch it names did.

Short names vs full names. You type main, but the true name is refs/heads/main. Git resolves a short name by searching a fixed order (refs/, refs/tags/, refs/heads/, refs/remotes/…), which is why a tag and a branch with the same name can be ambiguous. And to avoid thousands of one-line files, git periodically packs refs into a single .git/packed-refs — same refs, stored compactly.

Inspect refs yourself. git show-ref lists every ref and its hash · git for-each-ref does the same with formatting · git symbolic-ref HEAD prints the branch HEAD points at · git rev-parse main resolves a name to its commit hash · cat .git/HEAD shows the symbolic ref in the raw.

1 · A branch is just a movable pointer

A branch is not a copy of your files and not a container of commits. It is a single line of text in a file under .git/refs/heads/<name> holding one 40-character commit hash (plus a trailing newline — 41 bytes total). That is the entire branch.

# the whole "main" branch is this one file: .git/refs/heads/main → 4399030f3a7bb2148351d1465efb42ccd62ecb11 # 40 hex chars + newline = 41 bytes. That's it.

This is why creating a branch is instant and free: git branch feature just writes one more 41-byte file. And committing is equally simple — git writes the new commit object, then rewrites the current branch's file to hold the new commit's hash. The branch follows you forward.

"Branch" vs "the commits on it". People say "delete the branch and you lose the commits", but deleting a branch only removes the pointer file. The commit objects live in .git/objects and survive until garbage collection — if another ref (or the reflog) still reaches them, they're perfectly safe.

2 · HEAD — the pointer to "where you are"

HEAD answers the question "what is my working tree based on, and where will my next commit go?". Crucially, in the normal case HEAD does not name a commit directly. It is a symbolic ref: .git/HEAD contains the text ref: refs/heads/main. It points at a branch, which points at a commit.

# normal state — HEAD is a symbolic ref (one level of indirection): .git/HEAD → ref: refs/heads/main # HEAD → branch .git/refs/heads/main → 4399030… # branch → commit

So the chain is HEAD → main → commit. This double-hop is exactly why git commit on main moves main forward and HEAD seems to follow: git resolves HEAD to the branch main and advances that branch's file. HEAD never had to change — it still says ref: refs/heads/main, which now points one commit further along. In the diagrams the green HEAD → main tag shows this indirection; when HEAD names a branch the tag carries the branch name with it.

3 · Tags — pointers that don't move

A tag is also just a ref, living under .git/refs/tags/<name>. The difference from a branch is behavioural: committing never moves a tag. A tag pins a name to one specific commit forever (until you delete or force it). That's what makes v1.0 a stable label for a release.

4 · The detached HEAD — the whole story

Here is the state that confuses everyone. When you check out something that is not a branch — a raw commit hash, a tag, or a remote-tracking ref like origin/main — git has nowhere to write "you are on branch X". So it puts the commit hash directly into .git/HEAD. Now HEAD → commit, with no branch in between. That is a detached HEAD.

# attached: HEAD → branch → commit .git/HEAD → ref: refs/heads/main # detached: HEAD → commit (a raw hash, no "ref:") .git/HEAD → c6ef29126bb8bb8ddaeeb74bf634ced4fda08c21

Why detached commits are easy to lose. A commit is only "kept" if some ref can reach it. While detached, your new commit is reachable only through HEAD. The moment you check out a branch, HEAD moves off it and nothing points at that commit — it becomes orphaned, reachable only via the reflog until garbage collection deletes it for good.

Recovering and avoiding the trap

If you already left and panicked: the commit is almost always still in the reflog. git reflog shows it; create a branch on it to save it:

# find the lost commit's hash in the reflog… git reflog # …then plant a branch on it (or: git switch -c rescued <hash>) git branch rescued <hash>

Better: save it before leaving. While still detached, run git switch -c newbranch — this plants a branch pointer on your current commit and re-attaches HEAD to it, so the work is now on a real, named branch.

switch vs checkout. Modern git splits the overloaded checkout into git switch (move between branches) and git restore (restore files). git switch warns far more loudly and refuses to detach unless you ask explicitly with git switch --detach <commit>. Handy extras: git switch - jumps back to the previous branch, and git switch -c <name> creates-and-switches in one step.

5 · Walking the graph: ~ vs ^ vs ^2

Once you can name any commit relative to a pointer, you rarely need raw hashes. The shorthands trip people up because ~ and ^ look interchangeable but answer different questions.

FormMeans
HEAD  /  @@ is a plain alias for HEAD — where you are now.
HEAD^The first parent of HEAD. HEAD^ = HEAD^1.
HEAD^2The second parent — only meaningful on a merge commit. It is the tip of the branch that was merged in.
HEAD~1Go back 1 step along first-parent links. On a non-merge commit, HEAD~1 ≡ HEAD^.
HEAD~2Go back 2 first-parent steps (i.e. HEAD^^). It keeps following the first parent each hop — it never crosses into a merged side branch.
HEAD@{2}Not ancestry. This is the reflog: where HEAD was 2 moves ago in time. It can point anywhere — a different branch, an amended commit — and usually differs from HEAD~2.

The split only becomes visible on a merge. Below, M is a merge commit with two parents: C (first parent, the branch you were on) and S (second parent, the branch you merged in). Watch where each shorthand lands:

~ follows first-parents; ^n picks a parent. Mnemonic: tilde counts, caret chooses. ~N walks N generations up the first-parent line. ^N selects the N-th parent of one commit (and stays there). So HEAD~2 = "two commits back" while HEAD^2 = "the merged-in side at this merge". They are only equal by accident.
HEAD@{n} lives in a different dimension. HEAD~n walks the commit graph (parentage, fixed by history). HEAD@{n} walks the reflog (time, a per-clone diary of where HEAD has been). After a rebase or a bad reset, HEAD@{1} is your time-machine back to the previous tip — see the reflog explainer.

Check your understanding

Why does committing on main move both main and HEAD, even though git only rewrote one file?
Because HEAD is a symbolic ref pointing at the branch, not at a commit. .git/HEAD still literally says ref: refs/heads/main before and after the commit. Git only rewrote .git/refs/heads/main to hold the new hash. HEAD "moved" only because it resolves through main, which now points one commit further on. One file changed; the indirection did the rest.
You did git checkout abc123, made a commit, then ran git checkout main. Where is that commit now?
Orphaned. You were in detached HEAD, so the new commit was reachable only through HEAD. Checking out main moved HEAD onto the branch, and now no ref reaches your commit. It still exists in .git/objects and in the reflog — recover it with git reflog then git branch rescued <hash> — but it will be garbage-collected eventually if you don't. The safe move is to run git switch -c <name> before leaving.
On a merge commit M, what is the difference between M^2 and M~2?
M^2 is the second parent of M — the tip of the branch that was merged in (one hop, picking the side branch). M~2 is two first-parent steps back — it follows the first parent twice and stays on the mainline, never entering the merged branch. They land on completely different commits. Caret chooses a parent; tilde counts generations.
Is HEAD@{2} the same as HEAD~2?
No, and they usually disagree. HEAD~2 is ancestry — two commits back in the history graph, fixed for everyone with this history. HEAD@{2} is the reflog — where your HEAD pointer was two moves ago in time, which is local to your clone and depends on what you've been doing (switching branches, resetting, rebasing). One is about the graph; the other is about your recent activity.
