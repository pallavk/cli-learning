Checkout, switch & restore: one command that did too much

git checkout is the most overloaded command in git: for years one verb did several unrelated jobs — some that move where you are, one that overwrites your files. That collision is why checkout has scared people into losing work. In 2019 git split it into switch and restore; this page shows why, and when to reach for which.

The key insight: moving a pointer and rewriting a file are opposite operations that happened to share a keyword. Once you separate "change where I am" from "change what my files contain," every form of checkout falls into one of two clear buckets — and the modern commands name the bucket for you.

1 · The overload: four jobs, one word

Classic git checkout answered to all of these. Look at the last one — it's nothing like the others.

# one command, several unrelated jobs: git checkout main # 1. switch to a branch → moves HEAD git checkout -b feature # 2. create a branch + switch → moves HEAD git checkout a1b2c3d # 3. land on a bare commit/tag → moves HEAD git checkout -- file.txt # 4. discard edits to a file → rewrites your file!

Jobs 1–3 relocate a pointer (HEAD) and leave your committed work alone. Job 4 reaches into your working tree and overwrites a file — and never moves HEAD at all.

One term to pin down first: a detached HEAD (that's job #3). Normally HEAD points at a branch name, and the branch points at a commit — when you commit, the branch moves forward and HEAD rides along with it. But when you check out a bare commit or a tag, HEAD points straight at that commit with no branch in between. That's "detached": you're parked on a commit, not on any branch. It isn't an error — it's just that any new commits you make there belong to no branch, so they're easy to lose once you leave. We'll use the word a few more times below; the full picture is on the HEAD & refs page.

2 · The split (git 2.23, 2019)

Two opposite operations, one keyword — a famous footgun. Because git checkout -- file shared a name with "switch branch," a stray branch-name clash or a mistyped path could silently throw away edits that exist in no commit. Git's fix was to split the command by intent: git switch only ever moves HEAD; git restore only ever changes file contents. The verb now tells you which bucket you're in.
CommandOnly ever touches…Mental model
git switchHEAD / branches (where you are)"Take me somewhere else"
git restoreFile contents (working tree / index)"Make these files look like <X>"

3 · switch — move where you are

Switching moves HEAD to another branch and updates your working tree to match that branch's commit. Nothing in history changes — you've just relocated. Watch the teal HEAD pill travel.

git switch main # go to an existing branch (was: checkout main) git switch -c feature # create + switch (was: checkout -b) git switch - # jump back to the previous branch git switch --detach v1.0.0 # intentionally detach onto a commit/tag

That last one is the honest version of job #3: landing in a detached HEAD (defined above) is now something you ask for by name with --detach, instead of stumbling into it the moment you happen to check out a commit.

4 · restore — change file contents

Restoring rewrites files to match some source — the index, HEAD, or any commit — and leaves HEAD and every branch exactly where they are. To prove it: the graph below does not move when you restore. Only your files do.

git restore file.txt # discard unstaged edits (working tree ← index) git restore --staged file.txt # unstage, keep the edits (index ← HEAD) git restore --source=HEAD~2 app.py # pull ONE file from an older commit git restore --staged --worktree . # blow away ALL local changes — careful
restore is the one with no undo. Discarding working-tree edits throws away changes that live in no commit — so the reflog can't bring them back, unlike a bad switch or reset (which only move pointers the reflog records). Treat git restore file like rm on your uncommitted work.

Everything checkout did, in its modern, intention-revealing form. checkout still works and isn't deprecated — but new docs increasingly use the split commands.

6 · When to reach for which

Ask one question: am I changing my location or my files? If you want to be somewhere else — a different branch, a new branch, a specific commit — that's switch. If you want different file contents — undo edits, unstage, or grab a file from the past — that's restore. On git 2.23+ prefer the split forms: the verb states your intent, and you can no longer discard a file while reaching for a branch.
See also. switch is just HEAD moving between branches; switch --detach is how you land in a detached HEAD on purpose. restore --staged is the file-scoped cousin of reset, and the unstaging half of the staging area. A mis-switch is recoverable via the reflog — a mis-restore is not.
