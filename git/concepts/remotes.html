Remotes: origin/main is a cache, not the server

Git is distributed — every clone is a complete repository. A "remote" is just a nickname for another copy, and pointers like origin/main are your local memory of where that copy was the last time you spoke to it. Almost every push/pull confusion comes from forgetting that.

When you clone, you get the whole history plus a set of read-only pointers — remote-tracking branches — recording the server's state as of your last contact. They do not update by magic. fetch, pull, and push are the three commands that move commits (and these pointers) across the gap.

history both sides share commits only you have (ahead) commits only the remote has (behind)

1 · A remote is just a named URL

A remote stores nothing on its own — it's a name bound to a URL. origin is the default nickname your clone gives the place it came from. You can add as many as you like.

# git remote -v origin https://github.com/you/project.git (fetch) origin https://github.com/you/project.git (push) # add others: a teammate's repo, your fork, a deploy target… git remote add upstream https://github.com/org/project.git

What makes a remote useful is the set of pointers git keeps for it under refs/remotes/origin/ — one per branch on the server. Those are the remote-tracking branches.

2 · origin/main is a cached pointer

Step through a normal day: you commit, a teammate pushes, you fetch, you reconcile, you push. Watch which pointers move on each command — and notice that fetch moves only origin/main and never touches your work.

The one idea to keep. origin/main is not the server's branch — it's a read-only bookmark inside your repo saying "this is where origin's main was the last time I synced." It moves only when you run fetch (or push). Between syncs it goes stale, and that staleness is exactly what "behind" counts.

3 · fetch vs pull vs push

CommandDirectionWhat it changesTouches your files?
git fetchDownloadUpdates remote-tracking branches (origin/*) onlyNo — working tree and your branches untouched
git pullDownload + integratefetch, then merge (or rebase) into your current branchYes — advances your branch and updates files
git pushUploadSends your commits to the remote; advances origin/* on successNo locally; updates the server
pull is two commands in a trench coat. git pull = git fetch then git merge origin/main into your branch. That hidden merge is why a pull can suddenly mint a merge commit or drop you into a conflict. Prefer git pull --rebase to replay your commits on top (linear history), or look before you leap: git fetch, then git log main..origin/main to see what's incoming.

4 · Tracking: how a bare push knows where to go

A local branch can be linked to a remote-tracking branch — its upstream. Once set, git push and git pull need no arguments, and git status can report "ahead / behind" by comparing the two.

# first push of a new branch — set the upstream once with -u git push -u origin feature # now bare commands know the target, and status can count divergence: git status # "Your branch is ahead of 'origin/feature' by 2 commits." git branch -vv # every local branch + the remote-tracking branch it follows

Ahead is the number of commits you have that the remote-tracking branch doesn't; behind is the reverse. Both are measured against your possibly-stale origin/* pointer — so a git fetch first gives you an honest count.

See also. A pull that diverges resolves with a merge or a rebase — both linked pages show exactly what happens to your commits. Remote-tracking branches are ordinary refs, just kept under refs/remotes/. And once you've pushed a branch, opening a pull request is the next step.
