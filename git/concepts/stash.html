Stash: your work, parked as commits

A stash isn't a magic clipboard or a temp file. It's a tiny set of real commits stored on a hidden ref — so "stash" is just git committing your mess somewhere out of the way, then cleaning your tree.

You're mid-change and need a clean working tree — to switch branches, pull, or test something. git stash snapshots your uncommitted work as commit objects under a special ref called refs/stash, then resets your tree back to the last commit. Later you pop it and the changes flow back. Once you see that a stash entry is a commit, everything about it makes sense.

your real history / HEAD the index snapshot (staged state) the stash commit (your working tree) a dropped stash, now unreachable

1 · What git stash actually creates

Your history is A → B → C and main is at C. Your working tree has edits — some staged, some not. Watch what git stash push builds. The surprise: it makes two commits, and the working-tree one has two parents.

A stash is a commit with two parents. The stash entry W points at: parent 1 = the commit you were on (C), and parent 2 = a throwaway commit I capturing your index (staged) state. W's own tree is your full working directory. That's how one stash can faithfully restore both what you'd staged and what you hadn't. (Add -u and there's a third parent for untracked files.) The ref refs/stash points at W — that's the entire "stash".

2 · The stash is a stack

You can stash repeatedly. refs/stash keeps a reflog, so each entry is addressable by position — newest on top:

# git stash list stash@{0} # WIP on main: 4399030 fix login ← most recent stash@{1} # WIP on feature: a1b2c3d add chart stash@{2} # On main: experiment with cache # apply a specific one: git stash pop stash@{1}

Two things people trip on here. First, the stash is global to the repo, not per-branch — stash@{0} is the same entry no matter which branch you're on. The "WIP on main" text is just a label recording where it was made; you can pop it onto a different branch entirely. Second, stash@{1} is a stack position, not a time-travel ref like HEAD@{1} — popping stash@{0} renumbers the rest.

3 · pop vs apply vs drop

Restoring runs the stash backwards: git replays W's changes onto your working tree. Then the question is whether the entry stays in the stack.

CommandRe-applies changes?Keeps the entry?
git stash popYesNo — drops it on success (the usual choice)
git stash applyYesYes — leave it, e.g. to apply to several branches
git stash dropNoNo — discard an entry you don't need
git stash branch <name>Yes, onto a new branchNo — best fix when a pop would conflict
pop can conflict — and then it doesn't drop. If your tree moved on since stashing, replaying W may collide. Git leaves the conflict markers and keeps the stash entry (so you don't lose it), which surprises people who assume pop always cleans up. Resolve the conflict, then git stash drop by hand. When you expect trouble, git stash branch newbranch checks out the stash's original base and applies it there cleanly — no conflict at all.
Useful variants. git stash -u includes untracked files (the third parent); -a includes ignored files too · git stash -p stashes only hunks you pick · git stash -k (--keep-index) stashes everything but leaves your staged changes in place — handy to test exactly what you're about to commit · git stash show -p stash@{0} shows the diff a stash holds.
"I lost a stash." Because entries are commits, a dropped stash usually still exists until garbage collection. git fsck --no-reflogs | grep commit can surface dangling stash commits, and the reflog for stash remembers recent drops. Stashes are far more recoverable than their throwaway feel suggests.
