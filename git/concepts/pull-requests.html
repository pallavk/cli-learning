Pull requests: proposing changes across a boundary

A pull request isn't a git command — git has no git pull-request. It's a feature the forge (GitHub, GitLab, Bitbucket) adds on top: a page that compares two branches, runs CI, collects review, and offers a merge button. Underneath, it's pure git — a branch you pushed, waiting to be merged.

The whole flow is: branch → commit → push the branch to a remote → open a PR → review → click merge. The only part that's new versus everything else in this set is that last button — and which of its three forms you pick decides what your history looks like forever.

1 · What a PR actually is

A pull request is a forge feature, not a git one. When you push a branch, the server now has two branches that differ. The forge notices, and offers to open a request: "please pull my branch's commits into yours." That request page bundles the diff, a discussion thread, CI status, and approvals. Accept it and the forge runs an ordinary git merge on its side. Git itself never knew a "PR" existed.

Because it's just a pushed branch plus a merge, everything you already know about remotes, merge, and rebase applies directly.

2 · The lifecycle

You branch off main, make your commits, and push the branch. Pushing doesn't merge anything — it just puts your branch on the server so the forge can compare it and show a “Compare & pull request” button.

Now reviewers comment, CI runs, you push fixes (the PR updates automatically — it tracks the branch). When everyone's happy, someone clicks merge. That's where the three forms diverge.

3 · The three merge buttons

Which to pick? Squash is the common default — one tidy commit per PR, messy work-in-progress commits forgotten. Merge commit keeps the literal truth of how work happened (good for long-lived feature branches). Rebase gives a linear history that still preserves your individual commits — at the cost of rewriting their hashes.

After a squash or rebase merge, your local branch is a trap. The commits now on main have different hashes than the ones on your local feature branch — git no longer sees them as the same commits. Don't keep building on the old branch: delete it, then git switch main && git pull and branch fresh. Reusing a squash-merged branch is the number-one way people accidentally resurrect already-merged changes.

4 · The fork model

When you can't push to a project directly (most open source), you fork it — a server-side copy you own — and PR from there. The convention is two remotes: origin (your fork, you push here) and upstream (the original, you PR against it).

git clone https://github.com/you/project.git # origin = your fork git remote add upstream https://github.com/org/project.git git fetch upstream git switch -c feature upstream/main # branch off the real main # …work, then push the branch to YOUR fork… git push -u origin feature # now open the PR against upstream/main # keep your branch current as upstream moves: git fetch upstream && git rebase upstream/main
See also. Pushing the branch is plain remotes work; the merge button runs the same merge or rebase these pages animate. If a PR goes wrong after merge, revert (not reset) is the safe undo on a shared branch.
