Git, deeply

Interactive explainers for the git concepts that don't click from a cheatsheet. Each one animates what actually happens to commits, pointers, and your files — step by step.

Most git confusion comes from one thing: the commands describe what to type, not what moves. These pages show the moving parts — the commit graph, the branch pointers, HEAD, the index, your working directory — and let you step through each operation and watch them change.

Read them in this order if you're starting fresh: HEAD & refs first (everything else builds on pointers), then three trees & reset, then merge, rebase, worktree, and reflog last as the safety net that makes all the rest fearless. The everyday operations (cherry-pick, revert, stash, bisect) and the collaboration & deeper cuts (remotes, pull requests, tags, the staging area) below stand on their own — dip into them whenever you hit one.

How git merge really works→

Fast-forward vs --no-ff vs the three-way merge, the merge base, two-parent merge commits, and reading conflict markers.

Understanding git rebase→

Rebase doesn't move commits — it replays them as new commits onto a new base. Plus interactive rebase, --onto, and the golden rule.

Why it's hard: "rewriting history" sounds dangerous and magical. It's just copy-with-new-parent, but the hashes change.

05

git worktree→

One repository, many working directories — each checked out to a different branch, at once. No second clone, no stashing to switch context.

Why it's hard: it breaks the "one folder = one branch" assumption everyone starts with.
shared .gitno stashsame-branch rule
06

git reflog: your safety net→

Git records every move HEAD makes. So a bad reset, a botched rebase, or a deleted branch is almost always recoverable — here's how.

Why it's hard: it's invisible until you need it, and HEAD@{1} (reflog time) is easily confused with HEAD~1 (ancestry).
recoveryHEAD@{n}undo anything

Everyday operations — moving & undoing work

07

Cherry-pick→

Copy the changes of one specific commit onto your current branch — a fresh commit with the same diff but a new hash. Copy, not move.

Why it's hard: it looks like "moving" a commit, but the original stays put and the hash changes — and picking across branches you'll later merge can double-apply.
copy one commitnew hashbackport
08

Revert: undo by moving forward→

Undo a commit by appending its inverse as a new commit — history preserved, safe to push. The forward-moving opposite of reset.

Why it's hard: constantly confused with reset; reverting a merge commit needs -m and carries a re-merge gotcha.
inverse diffvs resetsafe undo
09

Stash: work parked as commits→

Park uncommitted work on a hidden ref and clean your tree. A stash entry is secretly a commit with two parents — once you see that, it all makes sense.

Why it's hard: it feels like a clipboard but it's commits; a pop that conflicts keeps the entry; and the stash is global, not per-branch.
refs/stashtwo parentspop vs apply
10

Bisect: binary-search a bug→

Find the commit that introduced a regression in log₂(N) tests — nine, not four hundred. Fully automatable with git bisect run.

Why it's hard: it's a workflow, not one command; it strands you in a detached HEAD while it runs; and it assumes the bug is monotonic.
binary searchbisect rungood / bad

11

Remotes & tracking branches→

What origin/main actually is — a cached pointer, not the live server — and how fetch, pull, and push move commits and pointers across the gap.

Why it's hard: fetch vs pull is invisible from the command, and origin/main looks live but is a stale local bookmark until you sync.
origin/main = cachefetch vs pullahead / behind
12

Pull requests→

A PR isn't a git command — it's a pushed branch plus a merge button. The fork → push → review → merge flow, and what merge-commit vs squash vs rebase each do to history.

Why it's hard: it feels like a git feature but it's a forge layer; the three merge buttons silently produce three different histories.
not a git command3 merge buttonsfork & upstream
13

Tags & releases→

A tag is a pointer that doesn't move when you commit — the fixed handle you put on a release. Lightweight vs annotated, why tags don't push by default, and the detached-HEAD checkout.

Why it's hard: it looks like a branch but is nailed in place, and tags silently don't travel with a normal git push.
doesn't moveannotated vs lightweightv1.2.3
14

The staging area→

The index is a complete, editable draft of your next commit — not a list of filenames. Covers git add -p, the two columns of git status, and the two diffs.

Why it's hard: add copies content now, not "later" — so the same file lands in both columns, and there are two diffs for two different gaps.
the indexgit add -pstaged vs unstaged
