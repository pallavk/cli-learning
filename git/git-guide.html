git Field Guide.

A practical reference for commits, branches, and remotes — with a quiz to test yourself.

Everything in git flows through three areas: the working tree (the files you edit), the staging area or index (the snapshot you are about to commit), and the repository (the committed history, with HEAD pointing at where you are). You move files forward with git add (working tree → index) and git commit (index → history). Most confusion in git melts away once this model clicks.

The mental model

A change travels through three places: the working tree (your files), the staging area / index (what git add has captured), and the repository of commits (what git commit records). HEAD points at the commit/branch you are currently on.

Starting a repository

git initCreate a new, empty repository in the current folder (makes a .git directory).
git clone <url>Copy an existing remote repo, including its full history. Sets up origin automatically.
git config --global user.name "You"Set the name stamped on your commits (do the same for user.email).

The everyday cycle

git statusSee what is staged, modified, or untracked. Your most-used command.
git add <file>Stage a file's current contents into the index. git add . stages everything below the current dir.
git commit -m "msg"Record whatever is staged as a new commit. git commit -am "msg" auto-stages tracked files.
git log --onelineBrowse history, one compact line per commit. Add --graph --all to see branches.
git diffShow unstaged changes (working tree vs index). git diff --staged shows what is staged.
git show <hash>Show a single commit's message and the patch it introduced.
Memory hook: git diff alone = "what have I changed but not staged?" — git diff --staged = "what am I about to commit?"

Branching basics

git branchList local branches; the current one is marked with *.
git switch <name>Move onto an existing branch (the modern replacement for git checkout <name>).
git switch -c <name>Create a new branch and switch to it (old form: git checkout -b <name>).
git merge <name>Bring another branch's commits into the current one.

Working with a remote

git remote -vList remotes and their URLs (origin is the default after a clone).
git fetchDownload remote changes without touching your working branch.
git pullfetch + merge in one step — updates your current branch.
git push -u origin mainUpload commits and set the upstream so later push/pull need no arguments.

.gitignore

A .gitignore file lists patterns of files git should not track — build output, secrets, node_modules, etc.

Gotcha: ignore rules only affect untracked files. If a file is already tracked, adding it to .gitignore does nothing until you stop tracking it with git rm --cached <file>.

Undoing things (the four tools)

The classic source of confusion. Match the tool to the area you want to change.

git restore <file>Discard unstaged changes — restore the working file from the index/HEAD.
git restore --staged <file>Unstage a file (pull it out of the index) but keep your edits. Same as the old git reset <file>.
git commit --amendReplace the most recent commit (fix the message or add forgotten changes). Creates a new hash.
git revert <hash>Create a new commit that undoes an earlier one. Safe on shared branches — nothing is rewritten.

git reset — the three modes

All three move the branch pointer (HEAD). They differ in what they do to the index and working tree.

git reset --soft HEAD~1Move HEAD back; keep changes staged. Great for re-committing or squashing.
git reset --mixed HEAD~1Default. Move HEAD and reset the index; changes stay in the working tree but become unstaged.
git reset --hard HEAD~1Move HEAD and discard everything — index and working tree. Destroys uncommitted work.
reset vs revert vs restore: reset moves HEAD (rewrites local history) · revert adds an inverse commit (safe for shared history) · restore only touches files, never HEAD.

Stashing

git stashShelve uncommitted changes and return to a clean tree.
git stash listSee the stash stack.
git stash popRe-apply the latest stash and remove it from the stack.
git stash applyRe-apply but keep the stash entry.

Merging & conflicts

git merge featureFast-forwards if possible; otherwise creates a merge commit with two parents.
git merge --no-ff featureAlways create a merge commit, even when a fast-forward was possible — preserves the branch's shape.
git merge --abortBail out of an in-progress merge and restore the pre-merge state.
Fast-forward: if your branch is a direct ancestor, git just slides the pointer forward — no merge commit. Resolve a conflict by editing the file, then git add it and git commit.

Branch & remote upkeep

git branch -d featureDelete a merged branch (-D to force-delete an unmerged one).
git remote add upstream <url>Add a second remote (common for forks tracking the original repo).
git pull --rebaseFetch, then replay your local commits on top — avoids an extra merge commit.
git clean -fdDelete untracked files (-f) and dirs (-d). Preview first with -n.

Tags

git tag v1.0Lightweight tag — just a name on a commit.
git tag -a v1.0 -m "release"Annotated tag — a full object with tagger, date, message (preferred for releases).
git push origin v1.0Push one tag (tags are not sent by a plain git push). --tags pushes all.

Rebasing

Rebase replays your commits onto a new base, giving a clean linear history — at the cost of rewriting commit hashes.

git rebase mainMove your branch's commits on top of main's tip.
git rebase -i HEAD~3Interactive rebase: pick, squash, reword, edit, drop, or reorder the last 3 commits.
git rebase --continueAfter resolving a conflict (and git add), proceed. --skip drops a commit; --abort bails.
Golden rule: never rebase commits you have already pushed to a shared branch. Rewriting public history forces painful reconciliation on everyone else. Merge preserves true topology; rebase rewrites for a linear story.

Cherry-pick, bisect, blame

git cherry-pick <hash>Replay one specific commit onto the current branch (-n to stage without committing).
git bisect startBinary-search history for the commit that introduced a bug: mark good/bad until git isolates it.
git blame <file>Show the commit, author, and date that last touched each line.

reflog — the safety net

git reflogA local log of every move of HEAD. Lets you recover commits "lost" by a bad reset, rebase, or amend.
git reset --hard <hash>Recover by resetting back to a hash you found in the reflog.
Detached HEAD: checking out a commit or tag points HEAD directly at a commit, not a branch. Commits made there belong to no branch and can be lost — create a branch (git switch -c <name>) to keep them.

Pushing power-tools

git push --force-with-leaseSafer force-push: refuses if the remote branch moved since your last fetch (won't clobber a teammate's work).
git push origin --delete featureDelete a remote branch (older form: git push origin :feature).
Refspecs: a refspec is <src>:<dst>. The default fetch refspec +refs/heads/*:refs/remotes/origin/* maps remote branches into your origin/* tracking refs; the leading + allows non-fast-forward updates.

The .git internals (in brief)

Everything in .git is content-addressed by SHA. Four object types:

blobRaw file contents — no name, no metadata. Identical content is stored once (deduplicated).
treeA directory: names mapping to blobs and sub-trees. A commit points to one top-level tree.
commitA tree snapshot + parent commit(s) + author/committer + message. Diffs are computed, not stored.
refA named pointer to a commit (branches and tags live under .git/refs). Moving a branch just rewrites its ref.

Submodules & squashing

git submodule add <url> libs/fooNest another repo, pinned to a specific commit (a gitlink), recorded in .gitmodules.
git submodule update --init --recursivePopulate submodules after cloning (or clone with --recurse-submodules).
git reset --soft HEAD~3 then git commitSquash the last 3 commits into one — their changes stay staged, ready to recombine.
git merge --squash featureCollapse a branch's work into staged changes (no merge parent recorded); commit it yourself.
